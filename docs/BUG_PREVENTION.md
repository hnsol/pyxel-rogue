# BUG_PREVENTION

バグ・不具合修正後に、原因と再発防止策を蓄積するログ。
現象リストを増やすのではなく、「なぜ埋めたか」から次の作業ルーチンへ落とす。

## 運用

- 不具合修正のたびに、まず「なぜ自分がそのミスを埋めたか」を検討する
- 「原因」は現象や直接ミスで止めず、最低1段深く「なぜその見落とし方・判断順・確認省略を選んだか」まで書く
- 直接原因と真因を分ける。真因には、壊れていた作業前提、誤った抽象化、未確認の境界、会話上の曖昧さのうち少なくとも1つを含める
- その原因を二度と起こさないための作業方策を決める
- 方策を現在のコードにも適用し、類似ケースを探す
- 必要な場合だけ、`原因` / `再発防止` / `今回の適用` として追記する
- `NOTES.md` はユーザーとの窓口、ここは作業者向けの再発防止ログとして分ける
- 現象別の対処記録だけで終わらせず、次回作業時に使える確認観点を書く
- 類似確認は `rg` とテストで行い、見つからない場合も「確認済み」と書く
- 作業前にこのファイルを読み、今回該当する見出しを選んで「再発防止」を作業チェックリスト化する。該当なしの場合は「該当なし」と明示する
- 完了前に、作業前のチェックリストを消化したか確認してから完了報告する

## 2026-05-12 UI状態を単純化しすぎた

### 原因

- UIの表示状態を「選択中」「復元中」「ack待ち」などの単語で把握したつもりになり、実際の描画単位へ分解しなかった
- 復元カーソルでは「色番号」と「主要パレット上の見た目」を分けて確認しなかった
- ack必須メッセージでは「マーカー」と「本文色」を同じ条件で扱い、折り返し行という表示単位を確認しなかった
- タイトル画面で一度直した観点を、ゲーム中UIの共通部品へ戻して横展開しなかった

### 再発防止

- UI不具合を直す時は、最初に「表示状態」と「描画単位」を分ける
- 1つの状態が複数の表示要素に出る場合、条件を要素ごとに分ける
- 色は番号だけでなく、主要パレットでの見た目と role 名を確認する
- 折り返し、複数列、複数タブ、専用パレットなど、表示が分割されるUIでは先頭要素だけでなく継続要素もテストする
- ある画面で直した観点は、同じ定数・同じ role・同じ状態を使う他画面へ `rg` で横展開する

### 今回の適用

- `rg UI_RESTORED_CURSOR_COL TITLE_RESTORED_CURSOR_COL` で復元色の横展開を確認
- `rg message_ack_pending first_part important` でack表示条件の横展開を確認
- Action menu / Item picker / 難易度選択の復元色をテストで固定
- ack必須メッセージは、1行・複数行・承認後表示をテストで分けた
- Action menu の `Take off` とキーボード `T` が同じ armor.c:take_off() 入口になるよう、武器を候補から除外する回帰テストを追加
- Rogue 5.4.4 `command.c` と Action menu の対応表を照合し、`w`/`W` 逆転を修正。Action名と直接キーの対応表テストを追加

## 2026-05-12 文言変更の確認入口を落とした

### 原因

- 文言を「表示テキストの差し替え」として扱い、どのゲーム状態・variant・保存データからその文言が選ばれるかまで追わなかった
- 通常Rogueの勝利とNyandorの勝利を、どちらも `result_flags == "winner"` で同じものとして扱い、`winner` が「結果」だけでなく「文脈別の称号」まで決めてしまう危険を見落とした
- Nyandorをタイトル・深度制限・オンライン取得の表示variantとしては意識していたが、スコア行に残すべき永続メタデータとして扱っていなかった。そのため、ローカル履歴やオンライン取得後の行では「今の起動variant」ではなく「その行がどのvariantで作られたか」を見る必要がある、という設計境界を確認しなかった
- `pyxel` / `pixel`、`victory` / `vitory` の誤キーは、欠損文字列を「別名吸収すればよい」と見てしまい、Web版では外部JSON、内蔵カタログ、古いbundleが別々の入口になることを先にモデル化しなかった
- ユーザー指定の語感が重要な箇所で、原文・Rogue2.Official・Nyandor固有世界観の優先順位を決める前に、こちらの仮表現で穴を埋めた

### 真因

- 指摘を受けた時、最初に「どの出力が間違っているか」へ飛びつき、「その出力はどの永続データから再現されるか」を調べる手順を省いた。文言修正でも、表示関数だけでなく保存entry、オンライン送受信、過去履歴、variant固定ビルドまでが仕様面になる、という認識が足りなかった
- `result_flags` のような既存フィールドを、意味の範囲を確認せずに便利な分類子として使い回した。`winner` はRogue 5.4.4の結果種別であって、Nyandorを含む全variantの称号を一意に表すフィールドではない。この「既存分類に新しい意味を押し込む」癖が、通常勝利文言をNyandorへ漏らした
- Rogue2.Official確認とNyandor固有文言検討を同じ作業として混ぜた。原作準拠が必要な通常Rogueと、派生variantとして自然な表現が必要なNyandorを分けて扱わなかったため、「出典に合わせるべき文言」と「新しく設計すべき文言」の境界を曖昧にした
- Webで欠損が出た時も、原因を「キーが足りない」へ狭めすぎた。実際の真因は、文言ソースが外部JSON・内蔵カタログ・Web bundleに分かれているのに、どれが実行時に優先されるかを確認するチェックが作業ルーチンに入っていなかったこと
- 最後に、ユーザーの「これでいいのか」という確認に対して、実装済みか未実装かを明確に切り分ける前に文言案だけ返した。これにより、仕様提案と実装状態の区別が会話上も作業上も曖昧になった

### 再発防止

- `NOTES.md` の文言指摘を処理する前に、対象入口をチェックリスト化する
- 結果画面・スコア・履歴・オンライン行の文言は、表示時の現在状態だけでなく、保存されたentryに必要な識別子があるか確認する
- 同じ `result_flags` を複数variantで使う場合、文言を分ける根拠を `result_flags` へ混ぜず、entry側にvariantなどの文脈を残す
- 既存フィールドを分類に使う前に、「このフィールドは原作の何を表すか」「variant差分を含めても意味が破綻しないか」「履歴として保存しても後から再現できるか」を確認する
- 文言作業では、出典準拠、variant固有設計、ユーザー案反映を別々の判断として書き出してから実装する
- 文言案だけを返す時は「未実装」と明示し、実装依頼に進んだら保存データと表示データの両方にテストを置く
- 文言キーは `rg` で正規キー、誤キー、近い綴りをまとめて確認する
- 日英、ログ、Inventory、HUD、結果画面、Web bundle のうち影響する入口をテストか bundle 確認で固定する
- ユーザーの文言案がある場合は原文寄せを基本にし、勝手に世界観語へ言い換えない
- `NOTES.md` を `Processed` へ移す前に、指摘箇条を再読して未対応入口が残っていないか確認する

### 今回の適用

- `rg "nyandor|vitory|victory|pixel|pyxel"` で誤キーと正規キーを確認
- `pixel.nyandor_cat_recovered`、`pixel.nyandor_vitory_line1`、`pixel.nyandor_vitory_line2` を別名吸収し、欠損表示に戻らないテストを追加
- 日本語 Inventory 名を `ねこ（ニャンダーの魔除けつき）` にし、ログ・勝利文言もねこ表記へ寄せた
- `web/pyxel-rogue.pyxapp` と `web/rogue/pyxel-rogue.pyxapp` に更新後のキー・文言が入っていることを `unzip -p` で確認
- `pixel.*` 誤キーを文言JSON・内蔵カタログへ直接追加し、Webが古い外部JSONを読んでも内蔵カタログへ戻る回帰テストを追加
- Rogue2.Official の英日対応を `rg` とスクリプトで照合し、上階メッセージ、睡眠、弱体化、hold monster の日本語を出典寄せにした
- 勝利スコア行は通常勝利とNyandor勝利で入口を分け、通常勝利は Rogue2.Official、Nyandor勝利はvariant付きスコア行のテストで固定した

## 2026-05-13 画面遷移時の古い表示状態を残した

### 原因

- 以前の修正で「次階に見えなければよい」をメッセージ本文の鮮度だけで解決しようとし、トースト表示を状態機械として読まなかった
- `draw_msgs()` は `msgs` / `msg_turns` だけでなく、前フレームの `msg_toast_visible_keys` と退場中の `msg_toast_expire_keys` を使って、古い行を数フレーム残す設計だった。この保持状態を「描画の副産物」と軽く見て、階移動時にクリアすべき本体状態として扱わなかった
- 「ログ履歴は残す」と「画面上の前階メッセージは消す」が同時に必要なのに、ログデータと表示キャッシュの責務を図式化せず、`msg_turns` を古くするだけで両方を満たしたつもりになった
- 回帰テストも文字列が描画されないことだけを見ており、表示継続を生む内部キーが空になったかを確認していなかったため、退場アニメ経由の再表示リスクを捕まえられなかった

### 真因

- UI表示を「関数を呼ぶたびに現在データから作られるもの」と思い込み、実際にはフレーム間で状態を持つコンポーネントだという前提に切り替えられていなかった。Pyxelの1フレーム描画に寄せた実装でも、見た目の安定化のためにキャッシュや退場アニメ状態を持つなら、それはゲーム状態と同じく遷移時の責務になる
- 「消す」という要求を、データ削除、表示抑制、履歴保持、アニメ停止のどれかに分解しなかった。ユーザーの意図は「ログ履歴を失う」ではなく「次階の画面に前階メッセージを出さない」だったのに、実装上の操作単位を先に決めてしまった
- 前にも直したはず、というユーザーの指摘を、再発した設計バグの警告として扱わなかった。既存テストがあるならなぜ漏れたか、過去の修正がどの状態だけを対象にしていたかを先に調べるべきだった
- テストをユーザー可視の文字列だけに寄せすぎ、可視状態を作る内部状態を検査しなかった。表示バグでは「今この1回で出ない」だけでは足りず、「次フレーム以降に復活させる状態が残っていない」ことまで固定する必要がある
- つまり真因は、UIを状態遷移として読む前に局所的な値変更で直せると判断したこと。これが、前回修正でも今回修正でも同じ種類の取りこぼしを生んだ

### 再発防止

- 画面遷移で消すべきUIは、データ本体、表示中キー、アニメ中キー、レイアウトキャッシュを分けて確認する
- 表示の持ち越しが起きるUIでは、描画関数内で参照される永続フィールドを `rg` で洗い出し、遷移時に残してよいものと消すものを分類する
- 「消えた」ことのテストでは、対象文字列だけでなく表示状態キーも空になったことを確認する
- ユーザーが「前も直した」と言った場合は、過去テストと過去修正範囲を読み、何が未検査だったかを再発防止ログに明記してから修正する
- UIの「消す」「閉じる」「戻る」「遷移する」は、1フレーム後、退場アニメ中、次の入力後の3点で残留状態を確認する

### 今回の適用

- 階移動時にログ履歴は保持し、`msg_toast_visible_keys` / `msg_toast_expire_keys` / トースト行数をクリアするようにした
- 階移動後に前階メッセージが描画されず、表示状態キーも空になる回帰テストを追加した

## 2026-05-15 オンラインプロファイルの半端状態を許した

### 原因

- `local_only=False` と `server_token` を別々の値として扱い、オンライン成立条件を正規化で一箇所に閉じなかった
- 登録APIの `ok` だけを成功条件にして、オンライン維持に必須の token が返ったかを保存前に確認しなかった
- スコア名、タイトルのモード表示、同期送信可否が別々の関数で判定されるのに、半端な profile を入れた時の横断テストがなかった

### 再発防止

- 永続プロファイルは「名前」「認証情報」「モード表示」を別々に見ず、正規化後の不変条件をテストする
- オンライン登録・連携は、`ok` だけでなく次回同期に必要なフィールドが揃った時だけ保存する
- 同期系の修正では、スコア保存名とオンライン判定が同じ profile から矛盾なく出ることを確認する

### 今回の適用

- token が空なら `local_only=True` に正規化するテストを追加
- 登録成功レスポンスに token が無い場合、オンライン profile として保存しないテストを追加
- `rg "online_profile|local_only|server_token"` で profile 更新入口を確認
- `local_only=True` の profile 名を `guest` に正規化し、登録画面キャンセルや local-only 確定で非guest名を保存しないテストを追加

## 2026-05-16 オンライン同期でvariant境界を混ぜた

### 原因

- ローカルスコア一覧を「同じユーザー名なら同期対象」と扱い、通常RogueとNyandorのvariant境界を同期前に絞らなかった
- オンラインPOST、cachedオンライン表示、ローカルbest表示がそれぞれ別入口で `load_score_entries()` を直接読んでおり、1箇所の確認で全体が守られると思い込んだ
- サーバ側の通常シートにNyandor由来らしい5F勝利行が混入していることを、実レスポンス確認まで見落とした

### 再発防止

- オンライン同期・ランキング表示でローカルスコアを混ぜる時は、ユーザー名より先にvariantで絞る
- `load_score_entries()` をオンライン系で直接使う場合は、同期POST、ローカルtab、cached合成、local-best補助表示の全入口を確認する
- variant混線の疑いがある時は、通常/派生variant両方の実サーバGETを確認し、dummyと実スコアの混在状態を見る

### 今回の適用

- 通常モード同期がNyandorローカルスコアをPOSTしない赤テストを追加
- 通常モードのcachedオンライン表示へNyandorローカルスコアを混ぜない赤テストを追加
- 通常/Nyandorの実サーバweekly GETでdummy行が返ることを確認
- 通常/Nyandorの実サーバweekly/season GETでdummy行が返ることを確認し、表示側で難易度メタデータを持たないdummy行だけが落ちていたことを切り分けた
- オンラインdummyは表示補助なので実スコアの難易度分離とは別扱いにし、Normal以外でも表示される回帰テストを追加した
