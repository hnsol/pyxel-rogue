# 品質方針

このリポジトリは、Rogue 5.4.4 の挙動を検証可能な底本として扱いながら、実装・構造・検証・文書は現代的な Python プロジェクトとして美しく保つことを目指す。

## 目標

- 40年後にも、仕様・設計判断・検証方法を追えるリポジトリにする
- 人間にも将来の機械的な読解にも、責務境界と変更理由が伝わる状態を保つ
- Python として自然で、テストしやすく、改訂しやすい構造にする
- 美しさを個人の感覚に閉じず、チェック・テスト・文書で維持できる土台にする

## Rogue 5.4.4 の位置づけ

Rogue 5.4.4 は挙動の底本であり、コード構造の模範ではない。

- ゲームメカニクス、確率、生成、戦闘、状態異常、アイテム効果は Rogue 5.4.4 C ソースを基準にする
- Python 実装は、原作の関数配置や global state をそのまま再現する必要はない
- 原作との差分が必要な場合は、表示・入力・移植都合なのか、ゲーム状態へ影響する差分なのかを分けて記録する
- テスト名、コメント、または `docs/DESIGN.md` から、参照した原作の関数・定数・テーブルへ辿れるようにする

## コードの美しさ

- 責務単位で分ける。行数削減や抽出そのものを目的にしない
- 名前は短さより意味を優先する。ただし既存の局所的な短縮慣習と衝突させない
- データで表せるものはテーブル化し、分岐を増やしすぎない
- ゲームロジック、入力、描画、永続化、オンライン通信、文言管理を不用意に混ぜない
- wrapper 比較だけのテストや、実装都合を固定するテストを増やしすぎない
- 小さすぎる抽出で読解コストが上がる場合は、単一ファイル内で明確に区切る方を選ぶ

## 分割の判断基準

新しいモジュールに分けるのは、次の条件を複数満たす場合に限る。

- そのモジュール名だけで責務が説明できる
- 呼び出し元の状態を大量に渡さずに済む
- 単体テストが仕様を表し、単なる wrapper 一致確認で終わらない
- Rogue 5.4.4 参照、Pyxel 表示、オンライン処理などの境界が明確になる
- 将来の変更時に「どこを見ればよいか」が減る

分けない方がよい例:

- 1つの小関数だけを置くための新規モジュール
- `Game` の内部状態を薄く包むだけのヘルパー
- 抽出後に引数が増え、元より読みづらくなる処理
- テストが「旧実装と同じ結果」しか確認できない処理

## テストの方針

- テストは仕様、原作準拠、再発防止、公開APIの互換性を守るために置く
- 実装の形だけを固定するテストは最小限にする
- バグ修正では、再発時に落ちるテストを先に確認する
- UIや画像は、ファイル存在だけでなく実際のロード・描画入口まで確認する
- 日本語切替、Pyxel Web、外部データ、保存データなど入口が複数あるものは入口ごとに確認する

## 文書の方針

- `README.md` / `README.en.md`: 外向けの入口
- `docs/DESIGN.md`: 仕様判断と設計判断の履歴
- `docs/TODO.md`: 未完了タスクと優先度
- `docs/BUG_PREVENTION.md`: 再発防止の作業ルール
- `docs/dev/`: 開発者向けの調査・品質方針・分割計画
- `AGENTS.md`: 作業時に常に参照する短い運用ルール

文書には、成果物本文と作り手の設計メモを混ぜない。公開されうる文書では、個人環境を特定できる絶対パスやホームディレクトリを書かない。

## 維持の仕組み

- 変更前に、目的が「美しい repo」にどう効くかを短く確認する
- 変更後に、不要関数・重複・直書き文言・責務混在が増えていないか確認する
- `tools/check_project_rules.py`、`ruff`、通常/日本語 unittest を品質ゲートとして扱う
- 長めの作業では、複数コミットに分け、各コミットが独立して説明できる状態にする
- 迷ったら、抽出より先に責務境界・テスト価値・文書上の位置づけを確認する
